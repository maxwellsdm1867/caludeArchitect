When to Use Rules Files vs Directory CLAUDE.md

Both scope instructions to specific parts of your codebase. The exam tests when each is appropriate.

Criteria.claude/rules/ fileDirectory CLAUDE.md
Scope Files matching glob pattern(s) — can span directories All files in that directory and below
Location .claude/rules/rule-name.md path/to/dir/CLAUDE.md
Targeting paths: ["**/*.test.ts", "**/*.spec.ts"] Implicit — everything in the directory
Best for File-type rules (all tests, all migrations, all configs) Package/module-specific rules (API layer, frontend, etc.)
Activation When Claude reads/edits a file matching the glob When Claude reads/edits any file in that directory

Rule File Anatomy

A rule file uses YAML frontmatter with a paths: field, followed by Markdown instructions:

---
paths:
  - "**/*.test.ts"
  - "**/*.spec.ts"
---
# Testing Standards
- Use describe/it blocks, not test()
- Mock external services, never call real APIs
- Name tests: test_<behavior>_<condition>_<expected>

Common Glob Patterns

PatternMatches
**/*.test.tsAll TypeScript test files at any depth
src/api/**Everything under src/api/
*.pyPython files in root only
**/*.pyPython files at any depth
migrations/**/*.sqlSQL migration files
💡 Key Insight: Rules files with glob patterns are more precise than directory CLAUDE.md for cross-cutting concerns. When you need all test files across the entire repo to follow the same conventions regardless of which package they are in, a single .claude/rules/testing.md with paths: ["**/*.test.*"] is cleaner than duplicating a CLAUDE.md in every directory.
⚠ Exam Trap: "Test files in different directories use different conventions." The wrong answer puts a CLAUDE.md in each test directory. The right answer is a single .claude/rules/ file with a glob pattern matching all test files. Rules files handle cross-cutting concerns; directory CLAUDE.md handles location-specific context.
