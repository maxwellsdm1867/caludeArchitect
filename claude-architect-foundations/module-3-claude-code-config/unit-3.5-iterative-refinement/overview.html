Unit 3.5 — Iterative Refinement Techniques

Task Statement 3.5 — Apply iterative refinement patterns to improve Claude Code output quality

⏱ 2 min read → then open the Lab

Choosing the Right Refinement Technique

Iterative refinement means feeding Claude's output back to improve it. The exam tests which technique matches which problem.

Problem SignalTechniqueHow It Works
Output doesn't match desired format/style Input/Output Examples Show concrete before/after pairs; Claude learns the transformation pattern
Output has logical errors or edge cases Test-Driven Iteration Write tests first, let Claude iterate until tests pass; tests act as the spec
Requirements are unclear or evolving Interview Pattern Ask Claude to ask clarifying questions before generating; elicits hidden requirements
First attempt is close but needs polish Targeted Feedback Point out specific issues in the output and ask for targeted fixes only

The Interview Pattern

Instead of guessing requirements, instruct Claude to ask questions first:

  1. Tell Claude to ask 3-5 clarifying questions before generating
  2. Answer the questions with specific requirements
  3. Claude generates with full context, reducing iterations

This front-loads context gathering and produces better first drafts.

Test-Driven Iteration Flow

  1. Write failing tests that define the desired behavior
  2. Ask Claude to implement until tests pass
  3. Add edge case tests, iterate again
  4. Tests become the living specification
💡 Key Insight: Test-driven iteration is the most reliable refinement technique because tests provide an objective, automated success criterion. Input/output examples and interviews rely on human judgment each iteration. Tests let Claude self-verify and iterate autonomously until the spec is met.
⚠ Exam Trap: "A developer keeps manually reviewing and correcting Claude's output through many rounds." The wrong answer is "use a more capable model" or "write better instructions." The right answer is test-driven iteration — write tests that encode the requirements, then let Claude iterate against the tests. This eliminates the manual review bottleneck.
