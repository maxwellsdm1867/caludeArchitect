The CLAUDE.md Hierarchy: Where Should Your Rule Live?

Claude Code merges instructions from three levels. The exam tests whether you pick the right one.

LevelFile LocationShared via Git?Use When
User ~/.claude/CLAUDE.md No — personal only Personal preferences (editor style, language prefs)
Project CLAUDE.md or .claude/CLAUDE.md at repo root Yes — team-wide Team standards (coding style, architecture rules, setup)
Directory subdirectory/CLAUDE.md Yes — scoped Package-specific rules (test conventions, API patterns)

Merge order: User + Project + Directory. All three are merged together. Directory-level CLAUDE.md is loaded when Claude reads or edits files in that directory.

Additional mechanisms:

MechanismPurposeKey Detail
@import Pull content from other files into CLAUDE.md Keeps CLAUDE.md DRY; imports are resolved at load time
.claude/rules/ Path-specific rules with glob patterns YAML frontmatter with paths: field; more granular than directory CLAUDE.md
/memory command Persist facts across sessions Writes to ~/.claude/ (user) or .claude/ (project) memory files
💡 Key Insight: The CLAUDE.md hierarchy is a prompt layering system. User-level sets personal defaults, project-level sets team standards, and directory-level adds scoped context. The @import directive keeps large configurations modular without duplication. Think of it like CSS specificity — more specific rules supplement (not replace) broader ones.
⚠ Exam Trap: "A new team member doesn't see your coding standards." If the rules are in ~/.claude/CLAUDE.md (user-level), they are not version-controlled. The fix is to move them to project-level .claude/CLAUDE.md or CLAUDE.md at the repo root. User-level is personal only.
