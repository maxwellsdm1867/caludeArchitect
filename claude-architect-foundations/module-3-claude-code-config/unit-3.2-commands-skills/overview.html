Commands vs Skills: Which Do You Need?

Both are reusable prompt templates, but they serve different purposes and live in different places.

AspectCommandsSkills
Location (project) .claude/commands/ .claude/skills/
Location (personal) ~/.claude/commands/ N/A
File format Markdown (.md) with optional YAML frontmatter SKILL.md with required YAML frontmatter
Invocation /project:command-name or /user:command-name Auto-loaded based on context or invoked via slash command
Arguments $ARGUMENTS placeholder in template Defined in YAML frontmatter
Context Runs in current conversation by default context: fork runs in isolated context (no conversation bleed)
Best for Simple, repeatable tasks (review, generate, format) Complex workflows needing isolation or specialized context

Command File Anatomy

A command is a Markdown file in .claude/commands/. The filename (minus .md) becomes the slash command name.

FeatureSyntaxPurpose
Arguments $ARGUMENTS Replaced with text after the slash command
Nested directories .claude/commands/test/unit.md Creates /project:test:unit
Personal commands ~/.claude/commands/my-cmd.md Available as /user:my-cmd in any project
💡 Key Insight: The context: fork setting in SKILL.md frontmatter is what makes skills fundamentally different from commands. A forked context means the skill runs without the current conversation history, preventing context contamination. This is critical for review tasks where you need an independent perspective.
⚠ Exam Trap: "A developer's personal command works on their machine but not for teammates." Commands in ~/.claude/commands/ are user-level, not version-controlled. Move to .claude/commands/ (project-level) for team sharing. This parallels the CLAUDE.md hierarchy trap from Unit 3.1.
