Unit 2.4 — Large Codebase Context Management

Task Statement 5.4 — Apply context management techniques for large codebase interactions

⏱ 2 min read → then open the Lab

Context Strategy Decision Table

Large codebase sessions degrade predictably. The exam tests whether you choose the right intervention for the degradation stage.

SignalDiagnosisIntervention
"The pattern we discussed" Stage 2 — vague references replacing specifics Write a scratchpad file with concrete names, paths, decisions
"Typical patterns for this type of project" Stage 3 — context fully degraded Use /compact or delegate to a fresh subagent
Session spans 50+ files across the codebase Scope exceeds single-context capacity Subagent delegation — each subagent handles a module
Crash or disconnect mid-session All in-memory context lost Scratchpad files survive; read them to resume

The pattern: externalize context to files before it degrades. Scratchpad files (e.g., PROGRESS.md) act as persistent memory that survives context compression, session crashes, and subagent delegation boundaries.

💡 Key Insight: The /compact command in Claude Code compresses conversation history to reclaim context space, but it is a lossy operation — specific details may be lost. Always write key decisions and findings to a scratchpad file before compacting. The scratchpad file survives compaction and can be re-read afterward.
⚠ Exam Trap: "Use a larger context window model" is a distractor for context degradation problems. A bigger window delays degradation but does not prevent it. The "lost in the middle" effect still applies — information in the middle of a 200K context is referenced less accurately than at the boundaries. Structural solutions (scratchpad, subagents) fix the root cause.
