1 — Tool Descriptions (TS 2.1)

Description Quality Spectrum

LevelExampleSelection
Minimal"Gets data"Ambiguous
Better"Gets customer data by ID"OK for simple cases
Production"Gets customer data by ID. Use for profile queries. Do NOT use for orders."Reliable

Production Description Pattern

  1. What it does (specific capability)
  2. When to use it (positive triggers)
  3. When NOT to use it (negative boundaries + alternative)
  4. What it returns (output shape)

Exam Rule

Unreliable tool selection → improve descriptions first. Not routing classifiers, not few-shot, not consolidation.

Split vs Consolidate

Split when tools have distinct trigger contexts. Consolidate when users always need both together and inputs are shared.

2 — Structured Errors (TS 2.2)

Error Response Fields

FieldPurpose
isError: trueSignals failure (MCP protocol)
errorCategoryRoutes recovery strategy
isRetryableRetry (true) or propagate (false)
retryAfterMsBackoff hint for retryable errors
userMessageSafe message for end users

Recovery Decision Tree

rate_limit / timeout → isRetryable: true, retry with backoff
not_found / auth → isRetryable: false, propagate to user
validation → isRetryable: false, ask user to correct input
internal → isRetryable: true (may be transient)

Trap

Plain error strings (“Error: not found”) are always wrong. The agent needs structured metadata, not human-readable text.

3 — Tool Distribution (TS 2.3)

The 4-5 Tool Limit

18 tools on one agent → selection degrades. Scope to 4-5 tools per agent. Use a coordinator to route between specialized agents.

Cross-Role Tools

High-frequency tools needed by multiple agents (e.g., verify_fact) → add to each agent. Avoids routing overhead.

tool_choice Modes

ModeBehaviorWhen
"auto"May skip tool callConversational
"any"Must call some toolPipeline step (any tool OK)
{"name":"X"}Must call tool XForced first step

Exam Rule

18+ tools + unreliable selection → scope to 4-5 per agent. Not “better descriptions” (partial fix), not “bigger model.”

4 — MCP Server Integration (TS 2.4)

Config Locations

FileScopeUse For
.mcp.jsonShared (version controlled)Team tooling
~/.claude.jsonPersonal (not shared)Experimental servers

Both discovered at connection time. All servers available simultaneously.

Secret Management

  • "API_KEY": "${API_KEY}" — correct (env var expansion)
  • "API_KEY": "sk-real-key" — wrong (hardcoded in VCS)

Community Servers

Review source code, check tool descriptions, verify secret handling, test with your agents before adopting.

Trap

“Share team tools via ~/.claude.json” is always wrong. Use .mcp.json for team tooling.

5 — Built-in Tools (TS 2.5)

Tool Selection Guide

NeedTool
Find files by nameGlob (path search)
Find text inside filesGrep (content search)
Read a known fileRead
Modify part of a fileEdit (unique match)
Create / rewrite a fileWrite
Run commandsBash

Key Distinctions

  • Grep = content search (find function definitions, imports, TODOs)
  • Glob = path search (find .py files, config files, test files)
  • Edit requires unique old_string — add context or use replace_all

Exploration Strategy

Glob (find key files) → Read (examine them) → Grep (search for specifics)
