Unit 4.2 — Structured Error Responses for MCP Tools

Task Statement 2.2 — Design structured error responses that enable intelligent recovery in agentic workflows

⏱ 2 min read → then open the Lab

Error Response Decision Tree

When an MCP tool fails, the structure of the error response determines whether the agent can recover intelligently or must escalate. The exam tests this decision tree.

Error FieldPurposeExample
isError: true Signals that the tool call failed MCP protocol field on tool result
errorCategory Classifies the failure type for routing recovery "authentication", "not_found", "rate_limit", "validation", "internal"
isRetryable Tells the agent whether retrying may succeed true for rate_limit, false for not_found
retryAfterMs Specific backoff hint for retryable errors 5000 (wait 5 seconds before retry)
userMessage Human-readable message safe to show the end user "The file was not found. Please check the path."

The pattern: local recovery first, propagation second. If the agent can retry or use a fallback tool, it should. If recovery is impossible (authentication expired, resource permanently deleted), propagate to the user with a clear message.

💡 Key Insight: The isError flag and isRetryable boolean are the two fields the agent needs to make the right decision. Without isRetryable, the agent either retries everything (wasting calls on permanent failures) or retries nothing (missing easy recoveries from rate limits). The errorCategory further enables category-specific fallback strategies.
⚠ Exam Trap: Returning a plain error string (e.g., "Error: file not found") instead of a structured error object is always the wrong answer. The exam tests whether you provide machine-parseable error metadata, not human-readable strings. Similarly, returning isError: false with an error message in the content confuses the agent — always set isError: true on failures.
