Unit 1.6 — Multi-Instance and Multi-Pass Review

Task Statement 4.6 — Design multi-instance and multi-pass review architectures

⏱ 2 min read → then open the Lab

The Review Architecture Decision

Two problems, two solutions:

Problem 1: Self-Review Bias

  • Who is reviewing the generated code?
    • Same session that generated it
      • Self-review bias — retains reasoning context, cannot question its own assumptions
    • New instance that sees only the code
      • Independent review — evaluates code on its merits, catches assumption errors

Problem 2: Attention Dilution in Large PRs

  • How many files in the PR?
    • Small (1-3 files)
      • Single-pass review is adequate
    • Large (many files)
      • Pass 1: Per-file local analysis (one request per file)
        Pass 2: Cross-file integration analysis (data flow, API contracts, shared state)

In a single-pass review of many files, early files get thorough review while later files get progressively less attention. Splitting into per-file passes ensures consistent review depth. The cross-file pass then catches integration bugs that no single-file review can see.

💡 Key Insight: The generating instance retains its reasoning context — it literally cannot question its own assumptions. A fresh instance that receives only the code (no conversation history, no generation reasoning) evaluates each decision on its merits and is far more likely to catch assumption errors.
⚠ Exam Trap: "Use extended thinking" and "use a bigger context window" do NOT fix self-review bias or attention dilution. Extended thinking gives more reasoning time with the same anchoring bias. A bigger context window holds more text but still allocates attention unevenly. The correct answers are: independent instance for bias, per-file passes for dilution.
