1 — Explicit Criteria (TS 4.1)

Vague-to-Precise Spectrum

LevelExampleQuality
L1 Vague"Flag bad stuff"Unusable
L2 Directional"Flag security issues"High FP rate
L3 Specific"Flag SQL injection, XSS, path traversal"Good
L4 Precise"Flag user input concatenated into SQL without parameterization"Production-ready

Exam Rule "Be conservative" is ALWAYS a distractor

Telling the model "be conservative" or "only flag high-confidence" is never the fix for false positives. It reduces recall without improving precision.

3-Step FP Fix Recipe

  1. Disable the high-FP category entirely
  2. Rebuild its criteria with explicit DO/DON'T rules
  3. Re-enable with new precise criteria

Quick Self-Test

"Does the prompt define DO-flag / DON'T-flag criteria?" — If no, rewrite before anything else.

2 — Few-Shot Prompting (TS 4.2)

When to Use Few-Shot

  • Inconsistent output format or style
  • Ambiguous tool selection across calls
  • Varied document formats (invoices, receipts, etc.)
  • High false positive rates despite clear instructions

Anatomy of a Good Example

ComponentPurpose
InputRealistic source material
ActionWhat the model should do
ReasoningWhy this action is correct

Key Not just input → output. Include reasoning.

How Many Examples?

2–4 targeted examples covering distinct scenarios. Diminishing returns past 5.

Trap Happy-Path-Only Examples

Only showing "clean" examples → model hallucinates on edge cases. Always include at least one negative/edge-case example.

Exam Rule

Few-shot examples > detailed instructions for consistency problems. If both are answer options, prefer few-shot.

3 — Structured Output / tool_use (TS 4.3)

tool_choice Decision Table

ValueBehaviorUse When
"auto"May return text (no tool call)Model decides if extraction needed
"any"Must call some toolMultiple schemas; unknown doc type
{"type":"tool","name":"X"}Must call tool XSpecific extraction required first

Schema Design Rules

  • Optional data → nullable field (prevents fabrication)
  • Extensible categories → enum + "other" + detail field
  • Ambiguous data → "unclear" escape valve in enum

What Schemas Prevent vs. Don't

PREVENTSDOES NOT PREVENT
JSON syntax errorsSemantic errors (totals don't sum)
Missing required fieldsValues in wrong fields
Invalid enum valuesFabricated nullable-should-be values

4 — Validation & Retry (TS 4.4)

Retry Decision Tree

Format error (malformed JSON, wrong type) → RETRY
Structural error (missing field, extra field) → RETRY
Semantic error (values conflict, totals wrong) → RETRY w/ self-correction
Info absent from source (data not in document) → DO NOT RETRY

Retry Pattern

Send back: original doc + failed extraction + SPECIFIC error description. Generic "try again" does not work.

Self-Correction Technique

Extract both calculated_total and stated_total side-by-side, with a conflict_detected boolean. Let downstream code resolve.

Systematic FP Analysis

Add a detected_pattern field to capture why the model flagged something — enables batch analysis of false positive root causes.

5 — Batch Processing (TS 4.5)

Decision Matrix

RequirementAPI Choice
Blocking / real-time neededSynchronous (Messages API)
Non-blocking / cost-sensitiveBatch API

Key Numbers

  • 50% cost savings vs. synchronous API
  • 24-hour processing window (max)
  • No latency SLA — may finish in hours, may take 24h

Constraints

  • No multi-turn tool calling within a batch request
  • Correlate results by custom_id

SLA Math

submission_frequency = sla_hours - 24. If you need results within 48h, submit every 24h.

Failure Handling

  • Resubmit only failed docs (match by custom_id)
  • Chunk oversized batches

Trap

"Usually finishes in hours" is NOT an SLA basis. Design around the 24h worst case.

6 — Multi-Pass Review (TS 4.6)

Self-Review Bias

Same session → retains reasoning → misses own mistakes. The model is biased toward confirming its prior output.

Fix: Independent Instance

Use a separate API call with no generation context. The reviewer must not see the generator's chain of thought.

Multi-Pass for Large PRs

PassScopeFocus
Pass 1Per-fileLocal analysis (bugs, style, logic)
Pass 2Cross-fileIntegration (data flow, API contracts, consistency)

Root Cause of Inconsistent Depth

Attention dilution — large input causes the model to skim rather than deeply analyze each section.

Traps (all WRONG fixes)

  • Bigger context window ≠ fix
  • Extended thinking ≠ fix
  • "Review carefully" instruction ≠ fix

8 — Distractor Detection Cheat Code

Common wrong answers on the exam. If you see these as options, eliminate them first:

Distractor PhrasingWhy It's WrongCorrect Direction
"Be conservative" / "only flag high-confidence" Reduces recall without fixing root cause Explicit DO/DON'T criteria (4.1)
"Add more detail to instructions" (when few-shot is an option) Instructions are not as effective as examples for consistency Few-shot examples (4.2)
"Use a bigger model" / "bigger context window" Does not fix attention dilution or structural problems Multi-pass architecture (4.6)
"Retry" (when info is absent from source) Can't extract what doesn't exist; retrying fabricates Nullable fields + accept absence (4.3/4.4)
"Batch API" (for blocking workflows) Batch API has no latency SLA; can't serve real-time Synchronous Messages API (4.5)
"Self-review in same session" / "extended thinking for review" Same context = same blind spots; thinking harder ≠ fresh perspective Independent review instance (4.6)
