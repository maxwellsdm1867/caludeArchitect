The Retry Decision Tree

Not all validation failures deserve a retry. Classify the error first:

  • Validation failed — what type of error?
    • Format mismatch
      • RETRY — high success rate (~90%+)

      e.g., "March 15, 2024" instead of "2024-03-15"

    • Structural error
      • RETRY — high success rate (~85%+)

      e.g., billing address placed in shipping field

    • Semantic error
      • RETRY with self-correction — medium success (~60%)

      e.g., line items don't sum to total

    • Information absent from source document
      • DO NOT RETRY — data doesn't exist

      Return null with a confidence flag instead

The key to effective retry: always include the specific validation error in the retry prompt — the original document, the failed extraction, and the exact error message. Never send a generic "try again."

💡 Key Insight: Include the specific validation error in the retry prompt — not a generic "try again." The model needs three things: (1) the original document, (2) the failed extraction, and (3) the exact error (e.g., "purchase_date '2024-13-01' is not a valid date — month cannot exceed 12"). Omitting any one cuts retry success significantly.
⚠ Exam Trap: Retrying when information is absent from the source document is wasted compute. The data does not exist — retrying just encourages the model to fabricate a value to "satisfy" the retry request. The correct action is to return null with a confidence flag.
