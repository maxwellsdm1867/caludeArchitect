Hub-and-Spoke: Coordinator-Subagent Architecture

The coordinator is the brain. Subagents are specialized workers. Communication always flows through the coordinator.

ComponentRoleContextTools
Coordinator Decomposes tasks, dispatches to subagents, synthesizes results Full conversation history Task tool (for spawning subagents)
Subagent Executes one focused task Only what the coordinator passes explicitly 4–5 domain-specific tools

Critical rule: subagents do NOT inherit the coordinator's context. Each subagent starts with a fresh conversation containing only the explicit prompt and context the coordinator provides. This isolation is a feature — it prevents context pollution and keeps subagents focused.

Design Principles

PrincipleRightWrong
Context isolation Pass only relevant context to each subagent Let subagents "see" the full coordinator history
Tool scoping 4–5 tools per subagent, domain-specific 18 tools on one agent (degrades tool selection)
Communication All results flow back through coordinator Subagents communicate directly with each other
Observability Coordinator logs all subagent dispatches and results Subagents operate as black boxes
💡 Key Insight: Subagents do NOT inherit the coordinator's context. This is the most frequently tested concept in Domain 1. Every subagent starts fresh — the coordinator must explicitly pass all relevant information. This isolation prevents context pollution and ensures each subagent stays focused on its specific task.
⚠ Exam Trap: "More tools = more capable agent" is always wrong. Research shows that giving a single agent 18+ tools degrades tool selection accuracy. The correct architecture: coordinator with a Task tool, subagents each scoped to 4–5 domain-specific tools. When you see an answer suggesting "add more tools to the agent," eliminate it.
