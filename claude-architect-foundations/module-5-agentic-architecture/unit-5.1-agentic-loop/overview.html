The Agentic Loop: stop_reason-Driven Control Flow

An agent is just a while loop. The model decides when to stop, not your code.

stop_reasonMeaningYour Code Does
"tool_use" Model wants to call one or more tools Execute tool(s), append results to messages, loop back
"end_turn" Model has finished — final response ready Break out of loop, present response to user
"max_tokens" Response hit the token limit Handle truncation (extend or surface partial result)

The correct pattern: while True → call API → check stop_reason → if "end_turn", break; if "tool_use", execute tools and continue. The model drives the loop, not arbitrary iteration caps or text parsing.

Anti-Patterns the Exam Tests

Anti-PatternWhy It Fails
Parsing natural language for termination
("I've completed the task")		 Model may say this mid-process; text content is unreliable as a control signal
Arbitrary iteration caps as primary stopping Cuts off legitimate multi-step reasoning; the model knows when it is done
Checking for text content as completion Model may emit text AND tool_use blocks in the same response
Pre-configured step sequences Removes the model's ability to adapt; agents should be model-driven
💡 Key Insight: The agentic loop is stop_reason-driven, not step-count-driven. The model signals completion via stop_reason: "end_turn". Your while loop should check this field — not parse text, count iterations, or follow a pre-configured sequence. Safety caps (e.g., max 25 iterations) are acceptable as a safety net, but never as the primary termination mechanism.
⚠ Exam Trap: Answer options that suggest checking for text content in the response (e.g., "if assistant message contains text, the task is complete") are always wrong. The model can return both text and tool_use blocks in a single response. Only stop_reason reliably indicates whether the model wants to continue or is done.
