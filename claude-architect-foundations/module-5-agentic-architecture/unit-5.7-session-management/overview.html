Unit 5.7 — Session State & Resumption

Task Statement 1.7 — Manage session state with --resume, fork_session, stale context risks, and fresh session + summary patterns

⏱ 2 min read → then open the Lab

Session Resumption Decision Table

MechanismContextUse WhenRisk
--resume Restores full prior session context Continuing an interrupted task with the same data Stale context if underlying data has changed
fork_session Snapshot of current context at fork point Exploratory branches, parallel approaches Divergent branches may conflict on shared resources
Fresh session + summary Clean context with structured summary of prior work When prior data may be stale, or for independent review Summary may lose nuance from the original session

The Stale Context Problem

ScenarioDanger of --resumeBetter Approach
Code review after PR updated Agent reviews outdated code from prior session Fresh session + summary of prior findings
Data analysis after DB migration Agent references old schema or data Fresh session with current schema
Debugging after a hotfix Agent's understanding of the bug is based on pre-fix state Fresh session + summary of what was tried and what changed
💡 Key Insight: --resume restores context but not reality. If the underlying data, code, or environment has changed since the session was paused, the resumed session operates on stale assumptions. The fresh session + summary pattern avoids this: start a new session, pass a structured summary of prior work, and let the agent re-examine current state.
⚠ Exam Trap: "Use --resume to continue the analysis" sounds efficient but is wrong when the underlying data has changed. If the exam scenario mentions updated files, new commits, or changed data since the session paused, the answer is fresh session + summary — not --resume. Stale context leads to incorrect conclusions.
