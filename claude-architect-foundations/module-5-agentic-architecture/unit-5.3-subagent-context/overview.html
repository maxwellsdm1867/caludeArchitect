Subagent Invocation Mechanisms

There are multiple ways to spawn and control subagents. Each has distinct context and tool behaviors.

MechanismContextToolsUse When
Task tool Fresh context; only the prompt you provide Scoped via allowedTools Default for coordinator-subagent dispatch
fork_session Snapshot of current session context Same as parent session Exploratory branches that share current understanding
Parallel Task calls Each gets independent fresh context Each scoped independently Independent tasks with no cross-dependencies

Context Passing: What to Include

IncludeExclude
Clear task description (what to do and what to return) Coordinator's full conversation history
Specific data the subagent needs (e.g., extracted text, URLs) Other subagents' internal reasoning
Output format requirements Unrelated prior results
Constraints and boundaries System-level configuration details

The allowedTools parameter restricts which tools a subagent can access. This is critical for security (prevent data exfiltration) and focus (prevent tool selection confusion).

💡 Key Insight: Context must be explicitly passed to subagents via the Task tool's prompt. The subagent cannot "see" anything the coordinator does not include. This means the quality of the coordinator's prompt to the subagent directly determines the subagent's effectiveness. Treat subagent prompts like function calls — specify inputs, expected outputs, and constraints.
⚠ Exam Trap: fork_session and Task are not interchangeable. fork_session inherits the current session's full context (useful for exploratory branches). Task creates a fresh context (useful for isolated subagent work). If an exam question asks about context isolation, the answer is Task, not fork_session.
