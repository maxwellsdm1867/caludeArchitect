The Tool Description Quality Spectrum

When tool selection is unreliable, the problem is almost always inadequate descriptions. The exam tests whether you improve descriptions before reaching for routing classifiers or few-shot examples.

LevelDescription StyleSelection Outcome
1 — Minimal "Retrieves customer information" Ambiguous with any tool that touches customer data
2 — Better "Retrieves customer account details by customer ID" Clearer input, but overlaps with order lookup when user mentions “my account order”
3 — Good "Retrieves customer account details by ID, email, or phone. Use for account/profile/membership queries." Positive trigger list guides selection but no negative boundary
4 — Production "Retrieves customer account details by ID, email, or phone. Use for account/profile/membership queries. Do NOT use for order lookups — use lookup_order instead." Positive triggers + negative boundary eliminates ambiguity

The pattern: a production tool description includes what the tool does, when to use it (positive triggers), and when NOT to use it (negative boundaries that point to the correct alternative).

💡 Key Insight: Tool descriptions are prompts. The same principles from Module 1 apply — explicit categorical criteria beat vague instructions. When two tools overlap, add a “Do NOT use for X — use Y instead” boundary to the description of each. This is the single highest-leverage fix for tool selection reliability.
⚠ Exam Trap: When a question describes unreliable tool selection, answer options like “add a routing classifier,” “use few-shot examples for tool selection,” or “consolidate tools into fewer endpoints” are almost always distractors. The correct first step is to improve the tool descriptions with explicit boundaries.
