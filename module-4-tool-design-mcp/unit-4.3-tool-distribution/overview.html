Task Statement 2.3 — Design tool distributions across agents and configure tool_choice for reliable selection

The 4-5 Tool Limit

Giving an agent 18 tools degrades selection reliability. The exam tests whether you scope each agent to 4-5 tools with optional cross-role tools.

Agent DesignToolsSelection Reliability
Monolith (wrong) 18 tools on one agent Degrades — too many options cause decision confusion
Scoped (correct) 4-5 per role-specific agent High — each agent has a focused, manageable toolset
Cross-role tool 1 shared tool (e.g., verify_fact) added to multiple agents High — high-frequency needs met without routing overhead

tool_choice Modes

ModeBehaviorUse When
"auto" Model decides whether to call a tool or respond with text Conversational agents where text responses are valid
"any" Model MUST call some tool (cannot respond with text only) Pipeline steps where tool execution is always required
{"type":"tool","name":"X"} Model MUST call the specific named tool Forced extraction, first step in a multi-step pipeline
💡 Key Insight: The 4-5 tool limit is about decision complexity, not capability. An agent with 18 tools can theoretically do more, but it selects the wrong tool more often. The fix is not better descriptions (though those help) — it is scoping each agent to a focused toolset that matches its role. Use a coordinator agent to route between specialized agents.
⚠ Exam Trap: When the question mentions 18+ tools and unreliable selection, the correct answer is almost always “scope to 4-5 tools per agent.” Distractors include “improve all tool descriptions” (partial fix, not root cause), “add a routing classifier” (unnecessary complexity), or “switch to a larger model” (model size does not fix decision complexity).
