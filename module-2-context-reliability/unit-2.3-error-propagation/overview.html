Unit 2.3 — Error Propagation in Multi-Agent Systems

Task Statement 5.3 — Design error handling that preserves context and enables intelligent recovery

⏱ 2 min read → then open the Lab

The Error Propagation Pyramid

When a subagent fails, the system's response determines whether the workflow recovers or collapses. The exam tests three patterns.

PatternBehaviorOutcome
Anti-pattern 1: Silent Suppression Subagent fails, returns empty results as "success" Coordinator proceeds with missing data — wrong answers, no warning
Anti-pattern 2: Catastrophic Termination Subagent fails, exception propagates up the chain Entire workflow terminates even when partial results are usable
Correct: Structured Propagation Subagent returns {error_type, attempted_query, partial_results, alternatives} Coordinator decides: retry, alternative approach, proceed with partial, or escalate

The critical distinction: access failures (permission denied, network timeout) are different from empty results (query returned zero rows). The first means "we couldn't look" — the second means "we looked and found nothing." Your error handling must distinguish these because the correct recovery action is different.

💡 Key Insight: Structured error context enables local recovery — the coordinator can retry with different parameters, try an alternative data source, or proceed with partial results and flag the gap. This is only possible when errors include error_type, attempted_query, and partial_results. Generic error messages ("something went wrong") force global escalation.
⚠ Exam Trap: "Wrap all subagent calls in try/except and continue on failure" looks like error handling but is actually silent suppression — the worst anti-pattern. The exam tests whether you recognize that catching and ignoring errors is worse than crashing, because it produces silently wrong results.
