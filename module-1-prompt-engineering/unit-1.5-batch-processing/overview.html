Sync API vs. Batch API

One question decides everything: is something blocked waiting for the result?

  • Is a user, developer, or pipeline waiting for the result before proceeding?
    • Yes — blocking workflow
      • Synchronous API — latency matters, pay full price

      e.g., pre-merge code review, PR comment generation, real-time chat

    • No — non-blocking workflow
      • Batch API — 50% savings, 24h window, no latency SLA

      e.g., nightly tech debt report, weekly security audit, training data classification

Batch API — Five Facts for the Exam

  1. 50% cost savings — the primary benefit.
  2. 24-hour processing window — no latency SLA whatsoever.
  3. custom_id — your correlation key; every request carries one, results come back tagged with it.
  4. No multi-turn tool calling — cannot execute a tool and feed results back within the same batch item.
  5. Failure recovery — resubmit only failed documents (by custom_id), not the entire batch.
💡 Key Insight: 50% savings, 24-hour window, no latency SLA, custom_id for correlation, no multi-turn. If the SLA deadline minus 24 hours is zero or negative, batch cannot be used at all — the SLA is tighter than the batch window allows.
⚠ Exam Trap: "Batch usually finishes in a few hours" is NOT an SLA. The exam will present someone arguing that batch is fine for blocking workflows because "it usually completes quickly." This is the #1 distractor. "Usually fast" is irrelevant — the 24-hour window is the only guarantee. Never use batch for blocking workflows.
