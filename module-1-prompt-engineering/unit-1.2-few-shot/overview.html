Unit 1.2 — Few-Shot Prompting for Consistency

Task Statement 4.2 — Apply few-shot prompting to improve output consistency and quality

⏱ 2 min read → then open the Lab

When to Use Few-Shot

Ask one question: is the output inconsistent across runs despite clear instructions?

  • Output inconsistent despite instructions?
    • Yes — format varies, tool selection unreliable, edge cases hallucinated
      • Add 2-4 few-shot examples with Input + Action + Reasoning
    • No — simple factual task, unambiguous output
      • Skip few-shot — adds tokens without value

Few-shot works because it demonstrates judgment. A rule says "handle ambiguous cases." An example shows what ambiguous looks like and what the correct response is — including why the alternative was rejected.

Each example must have three parts:

  1. Input — the actual user message or document
  2. Action — the tool call, classification, or extraction result
  3. Reasoning — why this action and why not the alternative

Without reasoning, examples are just a lookup table. The model matches on surface keywords instead of understanding the underlying logic.

💡 Key Insight: Examples must include Input → Action → Reasoning. Reasoning is what enables generalization to novel inputs. Without it, the model pattern-matches keywords ("cancel" → cancel_order) instead of understanding intent.
⚠ Exam Trap: Happy-path-only examples cause hallucination on edge cases. If all examples show clean, well-formatted inputs, the model has no guidance for messy real-world data and will invent values rather than return null. Always include at least one edge-case example with missing or ambiguous data.
